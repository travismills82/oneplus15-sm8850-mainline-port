# OnePlus 15 (SM8850) Linux 6.12 Mainline Port

This branch starts at the unmodified Linux **v6.12.23** tag and adds only the
porting/build automation. The primary build uses the kernel's normal Kbuild
interface (`make ARCH=arm64 LLVM=1`) and does **not** require Bazel.

## Source layout

The official OnePlus Android 16 release is split across three repositories and
uses branch `oneplus/sm8850_b_16.0.0_oneplus_15`:

- `OnePlusOSS/android_kernel_common_oneplus_sm8850` — Android common/GKI kernel
- `OnePlusOSS/android_kernel_oneplus_sm8850` — Qualcomm/OEM SoC kernel layer
- `OnePlusOSS/android_kernel_modules_and_devicetree_oneplus_sm8850` — modules,
  device trees, and the OEM Kleaf workspace

The Qualcomm target name in the OEM build configuration is `sun`. The common
kernel and upstream baseline both report 6.12.23, which makes this a useful
starting point, but equal version numbers do **not** mean the trees are source-
or ABI-compatible.

## First run

From the bootstrap bundle:

```bash
./bootstrap.sh --install-deps --sync-oem --build
```

From inside the generated kernel repository:

```bash
./tools/oneplus15-port/portctl doctor
./tools/oneplus15-port/portctl sync
./tools/oneplus15-port/portctl snapshot
./tools/oneplus15-port/portctl stock-images
./tools/oneplus15-port/portctl configure
./tools/oneplus15-port/portctl build

# After the clean baseline builds, merge compatible options from the stock config:
MERGE_STOCK_CONFIG=1 ./tools/oneplus15-port/portctl configure
./tools/oneplus15-port/portctl build
./tools/oneplus15-port/portctl oem-report
```

`stock-images` only **reads** boot-related partitions and saves them locally. No
provided command flashes the device.

## Branch and update model

- `oneplus15-6.12.y` — working mainline port branch
- `oneplus15-baseline-v6.12.23` — immutable marker at upstream v6.12.23
- `.port-work/refs.lock` — exact ACK and OnePlus source SHAs fetched by `sync`
- `rebase/v6.12.x` — generated by `portctl update-stable v6.12.x`

To move to a later 6.12 LTS release without rewriting the tested branch:

```bash
./tools/oneplus15-port/portctl update-stable v6.12.35
./tools/oneplus15-port/portctl configure
./tools/oneplus15-port/portctl build

# After the clean baseline builds, merge compatible options from the stock config:
MERGE_STOCK_CONFIG=1 ./tools/oneplus15-port/portctl configure
./tools/oneplus15-port/portctl build
```

The update command creates a backup branch and a new rebase branch. It does not
force-push or replace the tested port branch.

## Porting phases

1. Build an untouched mainline v6.12.23 arm64 kernel.
2. Capture the running phone's config, DT, command line, loaded modules, Kallsyms,
   and stock boot images.
3. Build the OEM common GKI reference with plain Kbuild where possible.
4. Compare OEM common against upstream and split required changes into small,
   reviewable subsystems.
5. Bring up early console, clocks, pinctrl, regulators, interrupt controller,
   interconnect, UFS, USB, and power management before higher-level hardware.
6. Port the OnePlus device tree and Qualcomm/OEM modules incrementally.
7. Preserve stock Android GKI/KMI compatibility as a separate goal from a true
   mainline boot. Many OEM modules cannot simply be loaded into an upstream
   kernel because they depend on Android-only APIs and a pinned KMI.

## What is automated now

- GitHub repository/fork creation through `gh`
- exact v6.12.23 baseline branch and marker tag
- official ACK and OnePlus source synchronization
- source SHA locking and OEM version checks
- rooted ADB inventory and read-only stock image backup
- stock-config filtering against symbols that exist in mainline
- non-Bazel LLVM/Kbuild compilation
- OEM common reference build attempt with Kbuild
- OEM/common-to-mainline change report and binary patch generation
- safe 6.12.y rebase branches
- GitHub Actions arm64 build verification

## Expected manual engineering

A script cannot automatically make all Android/Qualcomm vendor drivers suitable
for mainline. The first build proves the toolchain and upstream baseline. A
bootable phone kernel requires subsystem-by-subsystem code review, DT binding
conversion, upstream API adaptation, firmware validation, and repeated serial or
ramoops debugging. Camera, display, GPU, audio, WLAN, modem, and proprietary
power-management components are usually the longest phases.

Read `docs/SAFETY.md` before any boot or flash experiment.
